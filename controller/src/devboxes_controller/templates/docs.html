Devboxes documentation

From an idea to a ready workspace.

Devboxes gives you a prepared Linux machine on Kubernetes, reached from your terminal over SSH. This guide teaches the workflow you will use every day, and makes the storage boundaries explicit before anything destructive happens.

Terminal first SSH + tmux Persistent home Automatic compute stop

Quick start

Install the CLI once, authenticate it against {{ external_url }}, then create a medium box and enter it. The CLI waits until SSH is genuinely ready before connecting.

Install, authenticate, create, connect 
curl -fsSLO https://raw.githubusercontent.com/vicotrbb/devboxes/main/scripts/install-devbox-cli.sh
sh install-devbox-cli.sh
devbox login --url {{ external_url }}
devbox create atlas --preset medium --ttl {{ default_ttl_hours }} --repo owner/your-project --ssh
  1. Enter the access token once. The CLI verifies it and saves it in a user-only configuration file.
  2. Wait for “ready.” Kubernetes prepares the home volume, workspace pod, SSH service, and reachable address.
  3. Start working in ~/workspace. If you supplied --repo, the checkout is placed in ~/workspace/project.
  4. Detach instead of closing work. Press Ctrl+b, then d. Reconnect with devbox ssh atlas.
Your first mental shortcut

Closing SSH leaves tmux and your processes running. Stopping the devbox ends processes but keeps everything already written under /home/dev.

Know exactly what persists

A devbox has two lifetimes: disposable compute and a persistent {{ storage_class }} home volume. The following table is the source of truth for every lifecycle action.

Action Compute tmux and processes /home/dev How to return
Disconnect SSH Running Keep running Preserved devbox ssh <name>
Stop or expire Scaled to zero End Preserved devbox start <name>
Delete Removed End Retained Recreate the same name
Delete with --purge Removed End Permanently deleted Start from an empty box

Auto-stop behaves exactly like a manual stop. It never deletes a volume.

Create the right box

Names use 1 to 40 lowercase letters, digits, and hyphens. Choose a preset for the job and a TTL between 1 and {{ max_ttl_hours }} hours; {{ default_ttl_hours }} hours is the default.

Useful creation patterns 
# Empty box, connect when ready
devbox create scratch --ssh

# Clone a GitHub repository and use a larger workspace
devbox create compiler --preset large --ttl 72 --repo owner/compiler --ssh

# Provision asynchronously for later
devbox create nightly --preset small --no-wait

  • medium is the everyday default for compilers, agents, and local app stacks.
  • large is for memory-heavy builds, databases, or parallel tooling.
  • Reusing a retained volume can expand it for a larger preset; volumes are never shrunk.

Open a service running inside the box

Pass normal OpenSSH options after --. This forwards your local port 3000 to port 3000 inside the devbox while keeping the tmux session attached.

Forward a development server 
devbox ssh atlas -- -L 3000:127.0.0.1:3000

Then open http://127.0.0.1:3000 locally. Agent forwarding is also available with -- -A; use it only when the remote workflow needs your local agent.

GitHub, Codex, Claude, and installed tools

Workspace credentials arrive from a Kubernetes Secret at runtime. They are never baked into the image. GitHub tokens are loaded into the shell; Codex and Claude account files are seeded only when a persistent home does not already have newer login state. Every credential except the SSH public key is optional.

Confirm the prepared accounts 
gh auth status
git config --global --get-regexp '^user\.'
codex --version
claude --version
GitHub
Public repositories clone without a token. Add GH_TOKEN, author name, and author email to enable private clones and pushes.
Codex
Saved account state lives under ~/.codex. If it expires, run codex login --device-auth inside the box.
Claude Code
Saved account state lives under ~/.claude. Run claude and follow its sign-in prompt if a refresh is required.
User-installed tools
Cargo binaries, npm globals, and uv tool installations land in the persistent home and survive restart.

The daily workflow

Inspect and control boxes 
devbox list
devbox status atlas
devbox ssh atlas
devbox stop atlas
devbox start atlas
devbox delete atlas
devbox delete atlas --purge
Use the least destructive action

stop is for freeing compute. Plain delete removes the box but retains its home for reuse. Add --purge only when the files, history, configuration, host identity, and saved logins should disappear permanently.

Starting a stopped box renews its original TTL from the new start time. Deleting without purge removes the SSH address; recreating the same name adopts the retained home volume.

Use Devboxes from scripts

create, list, status, start, and stop support global --json. Use DEVBOX_TOKEN for non-interactive authentication so a token never appears in command history.

Find every ready box 
DEVBOX_URL="{{ external_url }}" \
DEVBOX_TOKEN="$TOKEN_FROM_YOUR_SECRET_STORE" \
  devbox --json list | jq -r '.[] | select(.state == "ready") | .name'
DEVBOX_TOKEN
Overrides the saved access token.
DEVBOX_URL
Overrides the saved API URL.
DEVBOX_CONFIG
Uses a separate config file for an isolated profile or test.
--url / --token
Highest-precedence one-command overrides; prefer the token environment variable for secrecy.

Use the browser workbench

The dashboard and CLI operate on the same Kubernetes resources. Use the browser for a quick fleet view, creation, lifecycle recovery, and copying an SSH coordinate; use the terminal when you are ready to work.

  • The fleet refreshes automatically every eight seconds; manual refresh is also available.
  • State is written as text, ready, starting, stopped, or degraded, rather than conveyed by color alone.
  • Deleting opens a confirmation that distinguishes retained storage from permanent purge.
  • The same access token creates a signed browser session; the raw token is not kept in the session cookie.

Troubleshooting

The CLI says authentication is required

Run devbox login --url {{ external_url }} and enter the access token created during installation. For a script, confirm DEVBOX_TOKEN is present and non-empty.

The TLS certificate is not trusted

Install the issuing certificate authority on the client machine or configure a publicly trusted certificate. The CLI deliberately has no insecure TLS switch; certificate failures should be fixed at the trust store.

A box stays in “starting”

Allow the first image pull and workspace initialization a few minutes. If it remains pending, inspect the dashboard message, SSH service address availability, and pod events.

The repository did not clone

Connect to the empty box, run gh auth status, then retry with gh repo clone OWNER/REPOSITORY ~/workspace/project. Confirm the shared GitHub token can read the repository.

SSH reports a changed host key after an intentional purge

A purged home creates a new host identity. Copy the install-scoped alias from the SSH warning, remove only that entry with ssh-keygen -R ALIAS, then reconnect and verify the new key.

Codex or Claude asks you to sign in again

For Codex, run codex login --device-auth. For Claude Code, run claude and follow the sign-in flow. The refreshed state is written to the persistent home.

Operator recovery

If the problem is below the CLI, inspect the Helm release, controller, generated box, and recent events.

Cluster diagnostics 
helm status devboxes -n {{ namespace }}
kubectl get deployment,service,pvc -n {{ namespace }}
kubectl logs -n {{ namespace }} deployment/devboxes --tail=100
kubectl get events -n {{ namespace }} --sort-by=.lastTimestamp

The shortest useful loop

Create, connect, detach, reconnect. Stop only when you want the compute back.
